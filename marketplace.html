Marketplace 开发与上架指南

DevToolBox · Marketplace 插件规范 / 开发流程 / 提交上架流程（详细版）

1. 总览

1.1 目标与边界

目标
  • 主程序与插件解耦：主程序保持稳定，插件独立迭代与上架。
  • 安全可控：插件在隔离环境运行，只能通过 SDK 使用被授权的能力。
  • 可审核与可治理：权限、域名、环境变量等信息在 manifest 中显式声明，便于审核。
边界
  • 插件不得直接调用 Electron/Node API，不得访问宿主文件系统真实路径，不得执行任意命令。
  • 宿主 SDK 是能力型接口，不提供“任意网络/任意文件/任意系统执行”。
  • 任何会触及“用户隐私/凭据/局域网”的能力都需要最小权限与明确说明。

1.2 运行模型

架构模型
  • iframe：插件 UI 运行在 iframe 隔离上下文。
  • Manifest：插件包根目录必须包含 manifest.json。
  • SDK：插件通过 postMessage 与宿主交互；宿主按权限进行拦截与执行。
Plugin (iframe)
  └─ postMessage (sdk request)
Host (renderer/preload/main)
  ├─ permission check
  ├─ capability execution (ipcMain handlers)
  └─ postMessage (sdk response)
兼容性
  • 插件使用 sdkVersion 声明与宿主 SDK 的契约版本。
  • 宿主应保持向后兼容：新增字段可选、禁用破坏性变更；必须破坏时提升 sdkVersion 主版本。

2.2 manifest.json 规范

字段说明（v1）
字段必填示例说明
id是market-mqtt-tool插件唯一标识，必须 market- 前缀
name是MQTT Tool展示名称
description是MQTT client utilities简述用途与价值

envAllowlist条件["API_BASE"]当声明 system:env:read 时必须提供
示例
{

  "name": "MQTT Tool",
  "description": "MQTT client utilities",
  "version": "1.2.0",
  "sdkVersion": "1.0",
  "entry": "package/index.html",
  "categoryId": "network-tools",
  "author": "DevToolBox",
  "license": "Apache-2.0",
  "homepage": "https://example.com",
  "repository": "https://example.com",
  "permissions": ["storage:kv", "http:proxy"],
  "httpDomains": ["iot.example.com", "*.example.org"]
}

2.3 权限与域名

最小权限原则
  • 只声明你实际需要的权限；不允许为了“将来可能需要”而提前申请。
  • 涉及网络的能力必须可追溯：httpDomains 必须准确、可解释。
  • 涉及环境变量读取必须白名单：envAllowlist 仅列出必要 key。
禁止项（治理约束）
  • httpDomains 不允许 *、不允许多级通配符 **.example.com、不允许带 scheme。
  • 宿主侧会阻止访问 localhost/私网/裸 IP（用于避免 SSRF 与内网探测）。
推荐权限清单（节选）
类别权限说明
Storagestorage:kv插件命名空间 KV，适合保存配置/历史记录
HTTPhttp:proxy调用 sdk.http.request 由宿主代发请求（可控、可审计）
HTTPhttp:external允许外部网络访问（需同时声明 httpDomains）
FSfs:dialog打开系统文件对话框（用户触发）
FSfs:read/fs:write基于 token 的读写（不暴露真实路径）
Systemsystem:getInfo读取系统信息（只读）

2.4 UI/交互规范

组件开发规范要求
  • 技术栈：React + TypeScript + Vite（插件模板已内置）。
  • 主题一致性：默认深色背景，避免硬编码过亮背景；按钮/边框保持对比度。
  • 可用性：关键操作必须可撤销或二次确认（删除/覆盖写入）。
  • 性能：避免一次渲染大量 DOM；长列表使用分页/虚拟化（必要时）。
  • 无障碍：按钮要有可读文本；图标按钮添加 title/aria-label。
推荐输出格式
  • 展示 JSON 使用格式化与高亮（必要时），避免输出一整行难以阅读的字符串。
  • 日志与结果区分：debug 走 sdk.log.*，用户可见结果写入 UI。

2.5 安全与合规

敏感信息处理
  • 不记录秘密：token、password、privateKey、session、cookie、证书私钥等不得写入日志。
  • 配置存储：用户敏感信息尽量不落盘；必须落盘时，应考虑加密/脱敏/提示风险。
  • 网络访问：严格限制域名；对用户输入 URL 进行校验，禁止内网/localhost。
发布包合规
  • zip 包内只能包含插件运行所需文件（manifest + entry root）。
  • 不得捆绑无关二进制、不得在包内携带密钥、不得包含企业内部 URL/凭据。

3. 开发流程

3.1 创建插件（模板）

一键生成（交互式）
./cli.sh plugin create

会在 marketplace/modules/<market-id> 创建基础模板（React + TS + Vite + 简化 SDK wrapper）。

模板默认行为
  • 入口：index.html + src/main.tsx。
  • 构建输出：vite build 到 package/（与 manifest.entry 对齐）。
  • SDK：通过 postMessage 调用宿主能力（示例包含 storage/system/notify）。

3.2 本地构建

构建插件
pnpm --filter @devtoolbox/plugin-<market-id> build
  • 构建成功后，必须确保 manifest.entry 指向的文件存在（默认 package/index.html）。
  • 插件代码应尽量将运行时依赖打包进 package/ 目录，避免运行时依赖宿主环境。

3.3 本地安装（file:// registry）

生成本地 registry 与 zip
node marketplace/scripts/pack-local.mjs <market-id>

会生成：

  • marketplace/registry.local.json
  • marketplace/.local-dist/<plugin>-<version>.zip
在 DevToolBox 中安装
  1. 启动：./cli.sh dev
  2. Settings → Marketplace Registry URL → 填入 file:///.../marketplace/registry.local.json
  3. Modules → Refresh → Install
注意
  • 开发环境允许 file:// registry；正式包仅允许 https://。
  • 如果你安装了历史版本插件导致行为异常，可使用 ./cli.sh clear 清理 marketplace 安装与缓存。

3.4 调试与日志

日志桥接
  • 插件日志推荐走 sdk.log.*，宿主会把它输出到 DevToolBox 控制台（必要时也会输出到 ./cli.sh dev 的终端）。
  • 不要把 token/密码/私钥等敏感信息输出到日志。
常用命令
./cli.sh dev
./cli.sh plugin <market-id>

./cli.sh plugin <market-id> 会引导你完成 build + 打包到本地 registry。

4. 上架流程

4.1 提交与审核

提交 PR 前必须完成
  • manifest 完整：必填字段齐全，permissions 最小化，httpDomains/envAllowlist 合规。
  • 安全自检：不包含 secrets；不访问 localhost/私网；不记录敏感日志。
  • 可复现构建：CI 能从源码构建出与上架一致的 zip。
  • 变更说明：版本号升级，说明新增能力、权限变化与兼容性影响。
审核关注点
  • 权限申请是否合理，尤其是 http:external/fs:*/system:env:read。
  • 域名白名单是否足够收敛，是否存在通配符过宽。
  • 是否包含敏感数据、内部地址、硬编码 token。

4.2 打包与发布（Marketplace Release）

发布产物
  • 每个插件一个 zip：<market-id>-<version>.zip
  • 一个总 registry：registry.json（列出所有插件的下载地址 + sha256 + size）
CI 打包脚本（仓库内）

CI 会使用 marketplace/scripts/pack-release.mjs 生成 release 目录：

GITHUB_REPOSITORY=owner/repo MARKETPLACE_TAG=marketplace \
node marketplace/scripts/pack-release.mjs
发布策略要点
  • downloadUrl 指向 GitHub Release 的 assets（由 tag 组织，例如 marketplace）。
  • 最终用户安装时拉取 registry.json，并校验 sha256 后再解压安装。

4.3 下架与回滚

治理操作
  • 紧急下架：将 registry 对应 entry 标记为 blocked（或移除），并在发布说明中标注原因。
  • 回滚：发布旧版本 zip 与 registry，或在 registry 中将推荐版本回退。
  • 兼容处理：若变更涉及 sdkVersion 或权限升级，需同步更新文档与审核项。

5. Checklist（作者自检）

功能与体验
  • 核心流程可用（无依赖隐含状态；断网/失败有提示）。
  • 重要操作可撤销或二次确认（删除/覆盖）。
  • 输出可读（JSON 格式化；长内容可复制）。
安全与合规
  • 不输出 secrets；不把敏感信息写入 storage。
  • 权限最小化；域名白名单收敛；不访问 localhost/私网。
  • release zip 不包含多余文件与二进制；不包含内部环境信息。
发布与治理
  • 版本号升级清晰（SemVer）；破坏性变更提高 sdkVersion。
  • manifest 字段齐全；entry 与构建产物一致。
  • 能用本地 registry 复现安装与运行。
